SDLC Multi-Agent System

Cross-platform workflow system for Claude Code, GitHub Copilot, and OpenAI Codex

0. What Is SDLC?

SDLC (Software Development Life Cycle) is a structured process for building software through defined phases — from understanding what to build, through designing and planning, to writing code and verifying quality.

Traditional SDLC involves humans in each role. This system replaces those roles with AI agents, each specialized for one job, while keeping a human in the loop at every phase gate.

Traditional SDLC
  • Analyst — gathers requirements
  • Architect — designs the system
  • Project Manager — breaks into tasks
  • Developer — writes code
  • QA Engineer — writes and runs tests
  • Code Reviewer — reviews quality
  • Security Engineer — audits vulnerabilities
This System (AI Agents)
  • Lead agent — clarify, research, design, plan
  • Lead agent — architecture decisions
  • Lead agent — task breakdown + dispatch
  • Coder agent — writes implementation code
  • Tester agent — writes and runs tests
  • Reviewer agent — quality verdict
  • Security agent — vulnerability scan
  • Rubber Duck agent — cross-model second opinion

3. SDLC Workflow Phases

1
Clarify
Refine scope, constraints, success criteria. Questions vary by workflow type.
⏸ USER APPROVAL → 00-clarify.md
2
Research
Analyze codebase: domain models, existing patterns, integration points.
⏸ USER APPROVAL → 01-research.md
3
Design
Produce architecture artifacts: diagrams, API contracts, storage model, testing strategy.
⏸ USER APPROVAL → 02-design/*.md
4
Plan
Break work into tasks by architecture layer. Each task gets a domain skill assignment.
⏸ USER APPROVAL → 03-plan.md
5
Implement
Multi-agent execution. For each task: Lead → Coder → Tester → Reviewer → Security → Rubber Duck (if enabled) → Lead (compliance).
⏸ USER APPROVAL per task → 04-implementation-log.md

6. Agent Context Isolation

Each agent sees only what it needs. Quality = (Correctness + Completeness) / (Context Size + Noise).

Coder Reads
  • task spec — single task from plan
  • api-contracts.md — API surface
  • storage-model.md — data layer
  • arch-decisions.md — constraints
  • domain skill — e.g. backend-node
Does NOT read: research, clarify, other tasks
Tester Reads
  • task spec — what was requested
  • coder's output — the code to test
  • testing-strategy.md — approach
  • domain skill — framework patterns
Does NOT read: arch-decisions, other tasks
Reviewer Reads
  • task spec — intent
  • coder's output — code changes
  • tester's output — test results
  • standard-verifications.md — checklist
  • principles.md — SOLID/KISS/YAGNI/DRY
Does NOT read: research, clarify, other tasks
Security Reads
  • task spec — scope
  • coder's output — code to audit
  • api-contracts.md — attack surface
  • error-handling.md — standards
Does NOT read: testing, clarify, other tasks
Rubber Duck Reads
  • full pipeline context — coder, tester, reviewer, security outputs
  • task spec — acceptance criteria
  • domain skill — same as Coder
Runs on opposite model. Skips what Reviewer + Security already caught.
